## Schema Changes

All additive. No breaking changes to existing tables.

### Topic status expansion

Add `review` to the allowed values for `topics.status`. New enum: `open`, `claimed`, `review`, `written`.

Update the `patchRecord` validator in `api.php` line ~193:
```php
['status' => ['open', 'claimed', 'review', 'written']]
```

### Calendar metrics columns

Added in the seed block (new DB creation) AND via migration check (existing DB):

```sql
-- In seed block: add columns to CREATE TABLE
-- For migration: check if column exists, ALTER if missing
ALTER TABLE calendar ADD COLUMN impressions INTEGER;
ALTER TABLE calendar ADD COLUMN clicks INTEGER;
ALTER TABLE calendar ADD COLUMN engagement_rate REAL;
ALTER TABLE calendar ADD COLUMN comments INTEGER;
```

Migration pattern (run after DB open, before routing):
```php
$cols = array_column($db->query("PRAGMA table_info(calendar)")->fetchAll(), 'name');
if (!in_array('impressions', $cols)) {
    $db->exec('ALTER TABLE calendar ADD COLUMN impressions INTEGER');
    $db->exec('ALTER TABLE calendar ADD COLUMN clicks INTEGER');
    $db->exec('ALTER TABLE calendar ADD COLUMN engagement_rate REAL');
    $db->exec('ALTER TABLE calendar ADD COLUMN comments INTEGER');
}
```

All nullable. NULL = no data (real or fake).

### Soft delete column

Add `archived_at DATETIME` to `submissions`, `calendar`, and `derivatives` tables. All nullable, default NULL.

Migration pattern same as above: check column existence, ALTER if missing.

This is a cross-cutting change. Every SELECT query in `api.php` (~15 queries) must add `WHERE archived_at IS NULL` (or `AND archived_at IS NULL` to existing WHERE clauses). Every DELETE handler becomes a soft delete:
```php
// Before: DELETE FROM submissions WHERE id = ?
// After:  UPDATE submissions SET archived_at = datetime('now') WHERE id = ? AND archived_at IS NULL
```

Affected endpoints: `DELETE /submissions/:id`, `DELETE /calendar/:id`.

### Derivatives table

```sql
CREATE TABLE derivatives (
    id INTEGER PRIMARY KEY,
    submission_id INTEGER NOT NULL,
    format TEXT NOT NULL,
    title TEXT NOT NULL,
    status TEXT DEFAULT 'generated',
    archived_at DATETIME,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (submission_id) REFERENCES submissions(id) ON DELETE CASCADE
);
CREATE INDEX idx_derivatives_submission ON derivatives(submission_id);
```

Allowed formats: `linkedin-post`, `blog-article`, `email-snippet`, `infographic-brief`, `social-carousel`, `newsletter-blurb`.

### Settings table

```sql
CREATE TABLE settings (
    key TEXT PRIMARY KEY,
    value TEXT
);
```

Stores `sheets_webhook_url` and any future config.

### Columns

| Column | Source | Note |
|--------|--------|------|
| Idee | `topics.status = 'open'` | |
| Schrijven | `topics.status = 'claimed'` | |
| Review | `topics.status = 'review'` | |
| Gepland | topic has calendar entry with `calendar.status = 'planned'` | One topic may have multiple calendar entries (LinkedIn + Dxfferent). Show in Gepland if ANY linked entry is `planned`. |
| Gepost | topic has calendar entry with `calendar.status = 'posted'` AND no `planned` entries remaining | Only when ALL linked entries are `posted` or `skipped`. |

### API

`GET /topics?view=kanban` returns topics with an additional `calendar_status` field. Implementation uses a subquery, not a simple JOIN, to handle one-to-many:

```sql
SELECT t.*,
  (SELECT GROUP_CONCAT(c.status) FROM calendar c 
   WHERE c.topic_id = t.id AND c.archived_at IS NULL) as calendar_statuses
FROM topics t
WHERE t.archived_at IS NULL
ORDER BY t.id
```

The JS parses `calendar_statuses` (comma-separated string or null) to determine which column a topic belongs to. Logic:
- If `calendar_statuses` contains `posted` and no `planned`: Gepost column
- If `calendar_statuses` contains `planned`: Gepland column
- Otherwise: use `topics.status` to place in Idee/Schrijven/Review

### Drag and drop

HTML5 Drag and Drop API (`dragstart`, `dragover`, `drop`). No library.

Drop actions:
- To Schrijven: PATCH topic status `claimed`, prompt name if unset
- To Review: PATCH topic status `review`
- To Gepland: open calendar modal pre-filled with topic, user picks date (creates calendar entry)
- To Gepost: PATCH linked calendar entry to `posted` (does NOT auto-generate metrics -- metrics stay NULL for real data)
- Backward moves (e.g. Gepland to Review): PATCH topic status to target status. Calendar entry is NOT deleted or archived -- it stays. The topic simply moves back in the workflow.

### Editor status button

The "Markeer als geschreven" button (`toggleDone()` in JS) currently toggles between `open` and `written`. Update to handle all statuses:

- If status is `written`: clicking sets to `open` (undo)
- If status is anything else (`open`, `claimed`, `review`): clicking sets to `written`
- Button label updates: "Geschreven" (green, when written), "Markeer als geschreven" (default, all other states)
- New: add a "Naar review" button next to it that sets status to `review`

### Card display

Same content as list view (title, category badge, meta) in narrower kanban column layout. Clicking a card in Schrijven/Review opens the editor. Derivative count shown as small `--ink-4` text indicator: "3 derivatives".

### Generation

Deterministic from `entry_id` as seed (use `$id * 2654435761 % range` pattern for reproducible pseudo-random). Account name hashed for per-consultant variation.

Generated during DB seed for `posted` entries ONLY. When a calendar entry is manually set to `posted` via the UI, metrics stay NULL (placeholder for real data).

A "Genereer demo data" button on the dashboard fills NULL metrics on `posted` entries with fake numbers. This button is only visible when there are `posted` entries with NULL metrics.

- "Content Performance": total impressions, avg engagement rate, total clicks. Uses aggregate SQL: `SELECT SUM(impressions), AVG(engagement_rate), SUM(clicks) FROM calendar WHERE impressions IS NOT NULL AND archived_at IS NULL`

- Per-consultant performance added to existing leaderboard card (avg engagement next to submission count)

### Calendar display

Posted entries with metrics show inline on the calendar card: "2.4K impressions, 4.2% engagement" in `--ink-4` color, below the existing meta line.

### Wipe

`POST /demo-data/clear` (not DELETE, since we're clearing values not deleting rows) sets all four metric columns to NULL and archives all derivatives.

## Feature 3: Content Derivatives

### Fake generation rules

Each seeded submission gets derivatives based on its topic category:

| Category | Derivatives |
|---|---|
| bewijs | linkedin-post, blog-article, email-snippet, infographic-brief (4) |
| educatie | linkedin-post, blog-article, social-carousel (3) |
| markt | linkedin-post, blog-article, newsletter-blurb (3) |
| contrair | linkedin-post, newsletter-blurb (2) |
| eigen | linkedin-post, blog-article, email-snippet (3) |
| tools | linkedin-post, social-carousel (2) |

Titles derived from parent topic title: "Case study: [topic]" for blog-article, "[first 80 chars]..." for linkedin-post, "Samenvatting: [topic]" for email-snippet, "Visual: [topic]" for infographic-brief, "Carousel: [topic]" for social-carousel, "Nieuwsbrief: [topic]" for newsletter-blurb.

All seed logic lives in `api.php` seed block. `seed.json` is NOT modified.

### API endpoints

- `GET /derivatives?submission_id=X` -- list derivatives for a submission (flat array)
- `GET /derivatives/tree?topic_id=X` -- nested JSON response built in PHP:

```json
{
  "topic": { "id": 1, "title": "..." },
  "submissions": [
    {
      "id": 5, "author": "Joshua", "created_at": "...",
      "derivatives": [
        { "id": 10, "format": "linkedin-post", "title": "...", "status": "generated" },
        { "id": 11, "format": "blog-article", "title": "...", "status": "generated" }
      ]
    }
  ]
}
```

This requires two queries (submissions for topic, then derivatives for those submission IDs), assembled into nested structure in PHP before `respond()`.

- `GET /derivatives/stats` -- aggregate counts by format: `SELECT format, COUNT(*) as count FROM derivatives WHERE archived_at IS NULL GROUP BY format`

### Board integration

Topic cards (both list and kanban) show "N derivatives" indicator when the topic has submissions with derivatives. Clicking the indicator expands an inline tree view below the card, styled with `--surface-2` background and `--ink-3` text. Tree uses CSS indent, no diagram library.

### Dashboard integration

New card "Content Derivatives" in `.dash-grid`: total derivatives by format as horizontal progress bars (same `.progress-row` pattern as existing domain/category bars).

### Wipe

`POST /demo-data/clear` archives all derivatives (`SET archived_at = datetime('now')`).


Default dates: first and last day of current month. Panel styled with `--surface-2` background, `--line` border.

### CSV format

```
date,title,platform,account,content_type,status,impressions,clicks,engagement_rate,comments
```

UTF-8 with BOM (`\xEF\xBB\xBF` prefix). Comma-delimited.

**Important**: The CSV endpoint must override the global `Content-Type: application/json` header set on `api.php` line 2. The endpoint outputs headers and body BEFORE the JSON header is written. Implementation: check for `/export/csv` route early in `api.php`, before the JSON Content-Type header, and exit with CSV output.

```php
// At top of api.php, before Content-Type: application/json
$pathCheck = $_SERVER['PATH_INFO'] ?? $_SERVER['ORIG_PATH_INFO'] ?? '';
if (str_contains($pathCheck, '/export/csv')) {
    // Handle CSV output with text/csv Content-Type and exit
}
```

API: `GET /export/csv?from=X&to=Y` -- returns file with `Content-Disposition: attachment; filename="content-calendar-YYYY-MM.csv"`.

### Google Sheets sync

- `POST /export/sheets?from=X&to=Y` reads calendar entries for the date range, POSTs JSON array to the Apps Script webhook URL (from settings table)
- PHP uses `file_get_contents` with stream context for the POST (no cURL dependency needed on shared hosting)
- Success/failure response returned to frontend, shown as toast
- If no webhook URL configured: button greyed out with "Stel eerst een webhook in"

Apps Script snippet (shown in UI when clicking "Webhook instellen"):
```javascript
function doPost(e) {
  var data = JSON.parse(e.postData.contents);
  var sheet = SpreadsheetApp.getActiveSpreadsheet().getActiveSheet();
  sheet.clear();
  sheet.appendRow(['date','title','platform','account','content_type','status','impressions','clicks','engagement_rate','comments']);
  data.forEach(function(row) {
    sheet.appendRow([row.date, row.title, row.platform, row.account, row.content_type, row.status, row.impressions, row.clicks, row.engagement_rate, row.comments]);
  });
  return ContentService.createTextOutput('OK');
}
```

- `POST /settings` -- write a setting (body: `{key, value}`)

## Demo Data Management

### Seed additions

The seed block in `api.php` (runs when `content.db` doesn't exist) is extended to:
1. Create all tables including `derivatives`, `settings` (with `archived_at` columns)
2. Add metrics columns to calendar CREATE TABLE
3. Insert 5 demo submissions (existing pattern)
4. Insert 15 calendar entries (existing pattern), with metrics on `posted` ones using deterministic generation
5. Insert derivatives for each demo submission per category rules

All seed logic lives in `api.php`. `seed.json` is NOT modified.

### Demo data clear endpoint

`POST /demo-data/clear`:
1. Set `impressions`, `clicks`, `engagement_rate`, `comments` to NULL on all calendar entries
2. Archive all derivatives (`SET archived_at = datetime('now')`)
3. Return `{cleared: true, metrics_cleared: N, derivatives_archived: M}`

Route: new case `demo-data` in the switch, with `$action === 'clear'`.

Does NOT touch topics, submissions, calendar entries, or settings.

### Full reseed

Delete `data/content.db` and hit any endpoint. Auto-seeds everything fresh. This is the only true destruction path.

## Architectural principle: no content destruction

No user content is ever hard-deleted. All delete operations use soft delete via `archived_at DATETIME` column. Existing `DELETE` endpoints on submissions and calendar become UPDATEs. All list/query endpoints filter `WHERE archived_at IS NULL`.

The demo data clear endpoint clears metric values and archives derivatives but never destroys submissions, calendar entries, or topics.

## Implementation constraints

- `seed.json` must NOT be modified -- all seed logic lives in `api.php`
- Auto-planner algorithm keeps working: it selects topics by `WHERE id NOT IN (scheduled topic IDs)` regardless of status. Auto-planned topics correctly appear in the Gepland kanban column because they create calendar entries with `planned` status.
- All new API endpoints follow existing patterns: same CORS, same `respond()`/`respondError()`, same `patchRecord()` helper
- CSV endpoint is the exception: must output before JSON Content-Type header (early exit pattern)
- Schema migrations use PRAGMA table_info check + ALTER TABLE for existing databases
- Deploy: SCP to SiteGround. For schema changes, either delete remote `data/content.db` for full reseed OR let the migration code handle it automatically
- Mobile responsive: all new UI works at 768px and below (kanban columns stack, drag replaced with dropdown)
- No hard deletes: all delete operations use soft delete via `archived_at`
- Single HTML file: no splitting, all CSS/JS inline per CLAUDE.md rules
