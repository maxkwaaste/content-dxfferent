# Phase 2.5 Design -- Proxuma Content Platform

**Date**: 2026-04-29
**Platform**: content.dxfferent.com
**Stack**: Vanilla JS + PHP + SQLite (no frameworks, no build step)
**Files**: `api.php`, `index.html`, `seed.json` (read-only)

---

## Schema Changes

All additive. No breaking changes to existing tables.

### Topic status expansion

Add `review` to the allowed values for `topics.status`. New enum: `open`, `claimed`, `review`, `written`.

Update the `patchRecord` validator for topics status field accordingly.

### Calendar metrics columns

```sql
ALTER TABLE calendar ADD COLUMN impressions INTEGER;
ALTER TABLE calendar ADD COLUMN clicks INTEGER;
ALTER TABLE calendar ADD COLUMN engagement_rate REAL;
ALTER TABLE calendar ADD COLUMN comments INTEGER;
```

All nullable. NULL = no data (real or fake). The seed populates these for `posted` entries only.

### Derivatives table

```sql
CREATE TABLE derivatives (
    id INTEGER PRIMARY KEY,
    submission_id INTEGER NOT NULL,
    format TEXT NOT NULL,
    title TEXT NOT NULL,
    status TEXT DEFAULT 'generated',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (submission_id) REFERENCES submissions(id) ON DELETE CASCADE
);
CREATE INDEX idx_derivatives_submission ON derivatives(submission_id);
```

Allowed formats: `linkedin-post`, `blog-article`, `email-snippet`, `infographic-brief`, `social-carousel`, `newsletter-blurb`.

Allowed statuses: `generated`, `reviewed`, `published`.

### Settings table

```sql
CREATE TABLE settings (
    key TEXT PRIMARY KEY,
    value TEXT
);
```

Stores `sheets_webhook_url` and any future config.

## Feature 1: Kanban Board

### Toggle UI

Two-button toggle in the Board header: "Lijst" (default, current view) / "Kanban". Persisted to `localStorage`. Existing filter pills (domain + category) apply to both views.

### Columns

| Column | Source |
|--------|--------|
| Idee | `topics.status = 'open'` |
| Schrijven | `topics.status = 'claimed'` |
| Review | `topics.status = 'review'` |
| Gepland | topic has calendar entry with `calendar.status = 'planned'` |
| Gepost | topic has calendar entry with `calendar.status = 'posted'` |

### API

`GET /topics?view=kanban` returns topics with an additional `calendar_status` field via LEFT JOIN on calendar. One query, no N+1.

### Drag and drop

HTML5 Drag and Drop API (`dragstart`, `dragover`, `drop`). No library.

Drop actions:
- To Schrijven: PATCH topic status `claimed`, prompt name if unset
- To Review: PATCH topic status `review`
- To Gepland: open calendar modal pre-filled with topic, user picks date (creates calendar entry)
- To Gepost: PATCH linked calendar entry to `posted`, generate metrics if demo mode
- Backward moves: PATCH topic status accordingly; if moving out of Gepland/Gepost, the calendar entry is not deleted (just the topic moves back)

### Card display

Same content as list view (title, category badge, meta) in a narrower layout. Clicking a card in Schrijven/Review opens the editor. Derivative count shown as small indicator.

### Mobile

Below 768px: columns stack vertically. Drag disabled. Each card gets a "Verplaats naar..." select dropdown.

| eigen | linkedin-post, blog-article, email-snippet (3) |

Titles derived from parent topic title with format-appropriate prefix.

### API endpoints

- `GET /derivatives?submission_id=X` -- list derivatives for a submission
- `GET /derivatives/tree?topic_id=X` -- full tree: topic > submissions > derivatives

## Feature 4: CSV + Google Sheets Export

### Export panel

Inline panel on calendar tab (not modal), triggered by "Export" button next to "Auto-plan 2 weken". Contains:

1. Two date inputs defaulting to current month
2. "Download CSV" button
3. "Sync naar Google Sheets" button
4. "Webhook instellen" settings link

### CSV format

```
date,title,platform,account,content_type,status,impressions,clicks,engagement_rate,comments
```

UTF-8 with BOM. Comma-delimited. Served from `GET /export/csv?from=X&to=Y` with `Content-Disposition: attachment` header.

### Google Sheets sync

- `POST /export/sheets?from=X&to=Y` reads calendar entries, POSTs JSON to Apps Script webhook URL (from settings table)
- Apps Script (provided as copy-paste snippet in UI): clears sheet, writes headers, writes rows. ~20 lines.
- Success/failure feedback via toast
- Button greyed out with "Stel eerst een webhook in" if no URL configured

### API endpoints

- `GET /export/csv?from=X&to=Y` -- CSV file download
- `POST /export/sheets?from=X&to=Y` -- Google Sheets push
- `GET /settings/:key` -- read a setting
- `POST /settings` -- write a setting (body: `{key, value}`)

## Demo Data Management

### Seed additions

The seed block in `api.php` (runs when `content.db` doesn't exist) is extended to:
1. Create new tables (derivatives, settings)
2. Add metrics columns to calendar
3. Insert 5 demo submissions (existing)
4. Insert 15 calendar entries (existing), with metrics on `posted` ones
5. Insert derivatives for each demo submission per category rules

### Wipe endpoint

`DELETE /demo-data`:
1. Set `impressions`, `clicks`, `engagement_rate`, `comments` to NULL on all calendar entries
2. Truncate derivatives table
3. Return `{wiped: true, metrics_cleared: N, derivatives_deleted: M}`

Does NOT delete topics, submissions, calendar entries, or settings.

### Full reseed

Delete `data/content.db` and hit any endpoint. Auto-seeds everything fresh.

---

## Architectural principle: no content destruction

No user content is ever hard-deleted. All delete operations become soft deletes via an `archived_at DATETIME` column on `submissions`, `calendar`, and `derivatives` tables. Existing `DELETE` endpoints set `archived_at = CURRENT_TIMESTAMP` instead of removing the row. All list/query endpoints filter `WHERE archived_at IS NULL`.

The "demo data wipe" (`DELETE /demo-data`) clears metric values (sets to NULL) and archives derivatives, but never destroys submissions, calendar entries, or topics.

Full reseed (deleting `content.db`) is the only true destruction path and is a deliberate admin action.

## Implementation constraints

- `seed.json` must NOT be modified
- Auto-planner algorithm must keep working (it doesn't filter by status, only by whether a topic has a calendar entry)
- All new API endpoints follow existing patterns: same CORS, same `respond()`/`respondError()`, same `patchRecord()` helper
- Deploy: SCP to SiteGround, delete remote `data/content.db` for schema changes, re-seed via API calls
- Mobile responsive: all new UI works at 768px and below
- No hard deletes: all delete operations use soft delete via `archived_at`
